## Overview

All ondoki configuration is done via environment variables. Copy `.env.example` to `.env` and customize for your deployment.

## Core

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `ENVIRONMENT` | — | `local` | Environment mode: `local`, `staging`, `production`, `test` |
| `DOMAIN` | Production | `localhost` | Domain for Caddy HTTPS (e.g., `app.ondoki.com`) |

## Database

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `POSTGRES_USER` | Yes | `postgres` | PostgreSQL username |
| `POSTGRES_PASSWORD` | Yes | `postgres` | PostgreSQL password |
| `POSTGRES_DB` | — | `ondoki` | Database name |
| `DATABASE_URL` | — | *(auto-composed)* | Full async connection string. Format: `postgresql+asyncpg://user:pass@host:5432/db` |

## Security

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `JWT_SECRET` | **Yes** | — | JWT signing key. Generate: `openssl rand -hex 32` |
| `ONDOKI_ENCRYPTION_KEY` | **Yes** | — | Fernet key for encrypting API keys at rest. Generate: `make generate-key` |

<Warning>
Never reuse these keys across environments. If you lose the encryption key, MCP API keys stored in the database cannot be decrypted and must be regenerated.
</Warning>

| `REDIS_PASSWORD` | Production | — | Redis password (set in production) |

## CORS & Frontend

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|

| `CORS_ORIGINS` | Production | `http://localhost:5173,ondoki://` | Comma-separated allowed origins |
| `CORS_ORIGIN_REGEX` | — | — | Regex pattern for CORS origin matching (e.g., `^https://.*\.ondoki\.com$`) |
| `ALLOWED_ORIGINS` | — | — | Additional allowed origins for CSRF checks (comma-separated) |

## SMTP (Email)

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `SR_SMTP_HOST` | — | *(empty)* | SMTP host. Leave empty to disable email |
| `SR_SMTP_PORT` | — | `587` | SMTP port |
| `SR_SMTP_USER` | — | — | SMTP username |
| `SR_SMTP_PASS` | — | — | SMTP password |
| `SR_FROM_EMAIL` | — | `noreply@ondoki.com` | Sender email address |

<Note>
Email is used for account verification and password reset. When `SR_SMTP_HOST` is empty, email features are disabled and users are auto-verified on registration.
</Note>

## Storage

Ondoki supports four storage backends. All file operations (workflow screenshots, uploads, exports) go through a unified abstraction — switch backends with a single env var.

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `STORAGE_BACKEND` | — | `local` | Storage backend: `local`, `s3`, `gcs`, or `azure` |
| `LOCAL_STORAGE_PATH` | — | `./storage/recordings` | Path for local file storage |
| `UPLOAD_DIR` | — | `./uploads` | Directory for general file uploads |

### S3-Compatible Storage (AWS S3, Cloudflare R2, MinIO, DigitalOcean Spaces, Backblaze B2, Wasabi, Hetzner)

Set `STORAGE_BACKEND=s3` and configure:

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `S3_BUCKET` | Yes | — | Bucket name |
| `S3_REGION` | — | — | AWS region (e.g., `eu-central-1`) |
| `S3_ENDPOINT_URL` | — | — | Custom endpoint for non-AWS providers (see examples below) |
| `S3_ACCESS_KEY_ID` | Yes | — | Access key |
| `S3_SECRET_ACCESS_KEY` | Yes | — | Secret key |
| `S3_PREFIX` | — | `uploads` | Key prefix for all stored objects |
| `S3_FORCE_PATH_STYLE` | — | `false` | Set to `true` for MinIO, LocalStack, and some R2 setups |

<Tip>
**S3_FORCE_PATH_STYLE** controls URL addressing:
- `false` (default) → `bucket.s3.amazonaws.com/key` (virtual-hosted, used by AWS)
- `true` → `s3.amazonaws.com/bucket/key` (path-style, required for most self-hosted S3)
</Tip>

**Provider examples:**

```bash
# AWS S3
S3_ENDPOINT_URL=             # leave empty for AWS
S3_FORCE_PATH_STYLE=false

# Cloudflare R2
S3_ENDPOINT_URL=https://<account-id>.r2.cloudflarestorage.com
S3_FORCE_PATH_STYLE=false

# MinIO (self-hosted)
S3_ENDPOINT_URL=http://minio:9000
S3_FORCE_PATH_STYLE=true

# DigitalOcean Spaces
S3_ENDPOINT_URL=https://<region>.digitaloceanspaces.com
S3_FORCE_PATH_STYLE=false

# Backblaze B2
S3_ENDPOINT_URL=https://s3.<region>.backblazeb2.com
S3_FORCE_PATH_STYLE=false

# Wasabi
S3_ENDPOINT_URL=https://s3.<region>.wasabisys.com
S3_FORCE_PATH_STYLE=false

# Hetzner Object Storage
S3_ENDPOINT_URL=https://fsn1.your-objectstorage.com
S3_FORCE_PATH_STYLE=false
```

### Google Cloud Storage

Set `STORAGE_BACKEND=gcs` and configure:

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `STORAGE_GCS_BUCKET` | Yes | — | GCS bucket name |
| `STORAGE_GCS_CREDENTIALS_FILE` | — | — | Path to service account JSON key file. If empty, uses Application Default Credentials (ADC) |

<Note>
When running on GCP (GKE, Cloud Run, Compute Engine), you can omit `STORAGE_GCS_CREDENTIALS_FILE` and use Workload Identity or the default service account instead.
</Note>

### Azure Blob Storage

Set `STORAGE_BACKEND=azure` and configure:

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `STORAGE_AZURE_CONTAINER` | Yes | — | Blob container name |
| `STORAGE_AZURE_CONNECTION_STRING` | Yes | — | Azure Storage connection string |

<Note>
Find your connection string in the Azure Portal under **Storage Account → Access keys**.
</Note>

## AI / LLM

LLM settings can also be configured from the **Project Settings → AI/LLM** page. UI settings take priority over environment variables.

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `ONDOKI_LLM_PROVIDER` | — | — | LLM provider: `openai`, `anthropic`, `ollama`, `copilot`, `custom` |
| `ONDOKI_LLM_API_KEY` | — | — | API key for the chosen provider |
| `ONDOKI_LLM_MODEL` | — | — | Model name (e.g., `gpt-4o`, `claude-sonnet-4-20250514`) |
| `ONDOKI_LLM_BASE_URL` | — | — | Base URL (required for `ollama` and `custom` providers) |

## PII Protection (SendCloak)

[SendCloak](https://github.com/myfoxit/sendcloak) automatically detects and replaces personal data before it reaches any LLM API. Responses are transparently deobfuscated.

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `SENDCLOAK_ENABLED` | — | `false` | Enable PII obfuscation before AI requests |
| `SENDCLOAK_URL` | — | `http://sendcloak:9090` | SendCloak service URL |
| `SENDCLOAK_STRATEGY` | — | `semantic` | Obfuscation strategy: `semantic` or `regex` |
| `PRESIDIO_LANGUAGE` | — | `en` | Default language for Presidio PII detection (if Presidio is enabled) |

<Tip>
SendCloak works standalone with built-in regex and pattern detection. For enhanced name/organization detection, add Microsoft Presidio:

```bash
# SendCloak only (fast, lightweight)
docker compose --profile privacy up -d

# SendCloak + Presidio (best detection accuracy)
docker compose --profile privacy --profile presidio up -d
```
</Tip>

<Note>
When `SENDCLOAK_ENABLED=true` but SendCloak is unreachable, AI features gracefully fall back to sending text without obfuscation (logged as a warning).
</Note>

## Generating Secrets

```bash
# JWT secret
openssl rand -hex 32

# Encryption key
python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
# Or use the Makefile:
make generate-key

# Strong password
openssl rand -base64 24
```
